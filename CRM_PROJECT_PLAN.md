## 1. Critical Rules

> These rules override everything else. Check before every commit without exception.

### 1.1 Commit Messages Are Single-Line Only

Every commit message must be exactly one line:

```
type: Imperative short message
```

**No body. No description block. No blank line after the subject. Nothing below the first line.**

Violating this causes multi-line commit messages to appear on GitHub, which looks unprofessional.

### 1.2 No AI Tool Attribution Anywhere вҖ” Including Commits

Never add `Co-Authored-By:`, `Generated by`, or any reference to Claude, ChatGPT, Copilot, or any other AI tool in:

- Commit messages
- Code comments
- README or documentation
- Any file that ends up in the repository

**Reason**: This is a university project. Any AI attribution visible on GitHub is an academic integrity violation.

## 2. Non-Negotiable Rules

### Language

**Everything in this project must be written in English.**
This includes: variable names, function names, comments, commit messages, README, error messages, and all other code artifacts. No exceptions.

### Commit Message Format

```
type: Imperative short message
```

| Type | When to use |
|---|---|
| `feat` | A new feature |
| `fix` | A bug fix |
| `docs` | Documentation changes only |
| `style` | Formatting, whitespace вҖ” no logic change |
| `refactor` | Code restructuring without behavior change |
| `test` | Adding or updating tests |
| `chore` | Build process, config files, dependencies, tooling |
| `perf` | Performance improvements |
| `ci` | CI/CD pipeline changes |
| `revert` | Reverting a previous commit |
| `build` | Changes to build system or external dependencies |

**Examples**:

```
feat: Add lead scoring algorithm
fix: Correct duplicate email validation
docs: Add API usage section to README
test: Add unit tests for scoringService
chore: Add ESLint configuration
refactor: Extract JWT verification into authMiddleware
style: Format route handler files
ci: Add lint step to GitHub Actions workflow
```

Rules:
- Message starts with a lowercase type tag followed by a colon
- The message after the colon is written in imperative mood (Add, Fix, Update вҖ” not Added, Fixed, Updated)
- No period at the end
- Keep it short and specific
- **One line only вҖ” no body, no description, no blank line after the subject**

### Comment Style

- Language: English only
- Tone: simple and technical вҖ” no conversational phrasing
- Scope: one short line per block describing **what it does**
- No multi-line comment blocks
- Bad: `// This function is responsible for calculating the priority score of the lead`
- Good: `// Returns 0-100 priority score based on engagement metrics`

## 7. Database Schema

```sql
-- UserAccount (NFR-ST-02: RBAC, NFR-ST-08: bcrypt hash)
CREATE TABLE UserAccount (
  user_id       SERIAL PRIMARY KEY,
  user_email    VARCHAR(48)  UNIQUE NOT NULL,  -- class diagram: String[48]
  user_password VARCHAR(255) NOT NULL,          -- bcrypt hash, plain text FORBIDDEN
  rbac_role     VARCHAR(20)  NOT NULL
                CHECK (rbac_role IN ('sales', 'support', 'admin')),
  full_name     VARCHAR(100) NOT NULL
);

-- Lead (FR-ST-01, FR-UC-02)
CREATE TABLE Lead (
  lead_id        SERIAL PRIMARY KEY,
  email          VARCHAR(100) UNIQUE NOT NULL,
  contact_name   VARCHAR(100) NOT NULL,
  priority_score DECIMAL(5,2) DEFAULT 0.0,      -- UC2: dynamic score (calculation вүӨ500ms)
  pipeline_stage VARCHAR(50)  DEFAULT 'New'
                 CHECK (pipeline_stage IN ('New', 'Contacted', 'Qualified', 'Closed')),
  user_id        INTEGER REFERENCES UserAccount(user_id),
  created_at     TIMESTAMP DEFAULT NOW()
);

-- SupportTicket (FR-ST-05, FR-ST-07)
CREATE TABLE SupportTicket (
  ticket_id      SERIAL PRIMARY KEY,
  description    TEXT NOT NULL,
  priority_level VARCHAR(10) NOT NULL
                 CHECK (priority_level IN ('Low', 'Medium', 'High')),
  status         VARCHAR(20) DEFAULT 'Open'
                 CHECK (status IN ('Open', 'In Progress', 'Resolved', 'Closed')),
  lead_id        INTEGER REFERENCES Lead(lead_id),
  user_id        INTEGER REFERENCES UserAccount(user_id),
  created_at     TIMESTAMP DEFAULT NOW(),
  updated_at     TIMESTAMP DEFAULT NOW()
);

-- InteractionLog (FR-ST-04)
CREATE TABLE InteractionLog (
  log_id    SERIAL PRIMARY KEY,
  note_text TEXT NOT NULL,
  timestamp TIMESTAMP DEFAULT NOW(),
  lead_id   INTEGER REFERENCES Lead(lead_id),
  user_id   INTEGER REFERENCES UserAccount(user_id)
);
```

### Table Relationships

```
UserAccount "1" в”Җв”Җв”Җв”Җ "N" Lead           : manages
UserAccount "1" в”Җв”Җв”Җв”Җ "N" SupportTicket  : handles
UserAccount "1" в”Җв”Җв”Җв”Җ "N" InteractionLog : authors
Lead        "1" в”Җв”Җв”Җв”Җ "N" SupportTicket  : has
Lead        "1" в”Җв”Җв”Җв”Җ "N" InteractionLog : collects
```

## 16. Development Guidelines

### 16.1 Layering & Separation of Concerns

Each layer has exactly one responsibility. Crossing the line is forbidden.

| Layer | Responsibility | Forbidden |
|---|---|---|
| `controllers/` | Parse request, call validator, call service, send HTTP response | Business logic, direct DB access |
| `services/` | Business logic, call repositories, return plain data | Touching `req`/`res`, sending HTTP responses, console.log |
| `repositories/` | Parameterized DB queries, return plain rows | Business logic, validation |
| `middleware/` | Token verification, role checking | Business logic, DB access |
| `db/` | Connection setup, schema loading | Business logic, validation |

### 16.2 Service Layer Conventions

- Functions return **plain objects/arrays** on success, **`null`** when an entity is not found.
- For business rule violations, throw a typed error with a stable code:

```js
const err = new Error('Duplicate email');
err.code = 'DUPLICATE_EMAIL';
throw err;
```

- Routes translate error codes into HTTP status codes.
- Never accept `req` or `res` as parameters.
- Functions stay under ~30 lines. One function does exactly one thing.

### 16.3 Route/Controller Layer Conventions

- Each handler stays under ~15 lines.
- Strict pattern: **validate вҶ’ call service вҶ’ respond**.
- All responses are JSON. Errors use this shape:

```json
{ "error": "ERROR_CODE", "message": "Human-readable description" }
```

### 16.4 Repository Layer Conventions

- **Always** use parameterized queries (`$1, $2, ...`). Never concatenate user input into SQL.
- Repositories only interact with the DB вҖ” no business logic, no validation.
- Return plain row objects or arrays.

### 16.5 Database Conventions

- Use `pg.Pool` for connection pooling.
- SSL required in production (`{ rejectUnauthorized: false }` for Render).
- Schema runs at startup from `schema.sql`.
- `JWT_SECRET` and `DATABASE_URL` only in `.env` and platform environment variables вҖ” never in source code.

### 16.6 Error Handling

- Services throw, controllers catch, return proper HTTP code + JSON error.
- Frontend: `api.js` throws on non-2xx; calling view catches and displays a message.
- Never swallow errors with empty `catch {}`.
- A single Express error-handling middleware sits at the bottom of `app.js` to catch unexpected throws and return `500`.

### 16.7 Naming Conventions

| Context | Convention | Example |
|---|---|---|
| JS variables, functions | `camelCase` | `calculateScore`, `leadId` |
| DB columns | `snake_case` | `priority_score`, `user_id` |
| Files (JS modules) | `camelCase.js` | `scoringService.js` |
| React components | `PascalCase.jsx` | `LeadForm.jsx` |
| Constants | `UPPER_SNAKE_CASE` | `MAX_EMAIL_LENGTH` |
| Error codes | `UPPER_SNAKE_CASE` | `DUPLICATE_EMAIL` |

### 16.8 Function and File Size

- Functions: under ~30 lines. Extract to a helper if you need an in-block comment.
- Files: under ~200 lines. If a service grows past that, split by responsibility.

### 16.9 Testing Patterns

- Unit tests mock repositories using `jest.mock()`.
- `scoringService` tests: pure data in/out вҖ” no mocking, no setup.
- One `describe` block per service module, one `test` per behavior.
- Tests must be deterministic вҖ” no real `new Date()`, no live network.
- At least one test per public service function, including edge cases.

### 16.10 Anti-Patterns (Strictly Forbidden)

- Business logic inside controllers
- DB queries inside controllers
- `req` or `res` reaching the service or repository layer
- Raw SQL string concatenation with user input
- Mutating function arguments
- Empty `catch {}` blocks
- Comments that restate what the code does
- Magic numbers (use named constants)
- Multi-line block comments
- Adding features beyond this plan without updating this document first
- Any reference to AI tools (Claude, ChatGPT, Copilot, etc.) in code, comments, commits, or docs

### 16.11 Plan-First Discipline

This document is the source of truth. If a decision arises that is not covered here, **update this document first**, then implement. Renaming an entity, adding a field, or changing an endpoint signature requires editing this file in the same commit.
