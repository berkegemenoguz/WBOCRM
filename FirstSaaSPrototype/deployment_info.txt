WBOCRM – Deployment Information
================================

Platform: Render (https://render.com)
Region:   Frankfurt (EU)

Services:
---------
1. Backend API
   Service name : wbocrm-backend
   Type         : Web Service (Node.js)
   Root dir     : backend/
   Build cmd    : npm install
   Start cmd    : npm start
   URL pattern  : https://wbocrm-backend.onrender.com

2. Frontend
   Service name : wbocrm-frontend
   Type         : Static Site
   Root dir     : frontend/
   Build cmd    : npm install && npm run build
   Publish dir  : dist/
   URL pattern  : https://wbocrm-frontend.onrender.com

3. Database
   Service name : wbocrm-db
   Type         : PostgreSQL (Render managed)
   SSL          : Required (auto-detected via DATABASE_URL hostname)

Environment Variables:
----------------------
Backend:
  DATABASE_URL  – Render injects automatically from linked DB
  JWT_SECRET    – Generate a strong random string (>= 32 chars)
  PORT          – Set to 5000 (Render also injects its own PORT)
  NODE_ENV      – production

Frontend:
  VITE_API_BASE_URL – https://wbocrm-backend.onrender.com/api

Deployment Steps:
-----------------
1. Push repository to GitHub
2. Go to https://render.com → New → Blueprint
3. Connect the GitHub repository
4. Render reads render.yaml and provisions all three services
5. After first deploy, run the schema:
   render run --service wbocrm-backend -- node src/db/init.js
6. Seed demo data (choose one method):
   a. CLI:  render run --service wbocrm-backend -- node src/db/seed.js
   b. API:  POST https://wbocrm-backend.onrender.com/api/setup
      (one-time endpoint that creates 3 users, 5 leads, 2 tickets)

Notes:
------
- Free tier services spin down after 15 minutes of inactivity; first
  request after spin-down takes ~30 seconds (cold start).
- SSL is enforced by pool.js: detects "render.com" in DATABASE_URL
  and enables { ssl: { rejectUnauthorized: false } }.
- JWT tokens expire after 30 minutes (see authService.js).
- All CORS origins should be updated in src/app.js for the live URL.
- The archive cron job starts automatically when the backend boots
  (archiveService.scheduleArchiveJob). It runs daily and archives
  resolved/closed tickets older than 365 days to ArchivedTicket table.
- 30-minute idle session timeout is enforced on the frontend via
  AuthContext (activity-based reset on mousemove/keydown/scroll).
